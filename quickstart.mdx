This guide will get you from installation to your first safe mutation workflow in under 5 minutes.

## Prerequisites

Make sure you have CURD installed. See the [installation guide](/installation) for detailed instructions.

<CodeGroup>

```bash Homebrew
brew tap aerospeedsta/curd && brew install curd
```

```bash Winget
winget install curd
```

```bash Python (uvx)
uvx --from curd-python curd --version
```

</CodeGroup>

## Initialize Your Workspace

<Steps>

<Step title="Navigate to your project">
```bash
cd /path/to/your/project
```
</Step>

<Step title="Initialize CURD">
```bash
curd init
```

This creates a `.curd/` directory and detects:
- Source layout (language, file structure)
- Build system (Cargo, npm, Maven, etc.)
- Dependency ecosystem
- Plugin-derived metadata (if enabled)
</Step>

<Step title="Verify the setup">
```bash
curd doctor .
```

The doctor command runs diagnostics to ensure everything is configured correctly.
</Step>

</Steps>

## Explore Your Codebase

Now that CURD is initialized, let's explore the code using the three core tools.

<Steps>

<Step title="Search for symbols">
```bash
curd search alpha
```

This performs BM25-ranked semantic search across your codebase. Results are ranked by relevance and show:
- Symbol name and type
- File location
- Context snippet
</Step>

<Step title="Read a specific symbol">
```bash
curd read src/lib.rs::alpha
```

The URI format is `file::symbol` or `file::symbol::line`. This returns AST-aware boundaries around the symbol.
</Step>

<Step title="Analyze dependencies">
```bash
curd graph src/lib.rs::alpha --direction both --depth 2
```

This shows:
- **Up**: What calls this symbol (callers/importers)
- **Down**: What this symbol calls (callees/dependencies)
- **Both**: Complete neighborhood view
- **Depth**: How many hops to traverse
</Step>

</Steps>

<Note>
These three commands—`search`, `read`, and `graph`—are read-only and safe to run anytime. They form the foundation of CURD's code intelligence layer.
</Note>

## Your First Safe Edit

CURD requires a shadow workspace session before making any mutations. This ensures you can inspect, test, and rollback changes before committing.

<Steps>

<Step title="Begin a workspace session">
```bash
curd workspace begin
```

This creates a physical shadow workspace at `.curd/shadow/root` where all changes are staged.
</Step>

<Step title="Edit a symbol">
```bash
curd edit src/lib.rs::alpha --code "pub fn alpha() { println!(\"Updated\"); }"
```

The edit is applied to the shadow workspace, not your actual files yet.
</Step>

<Step title="Inspect the diff">
```bash
curd workspace diff
```

This shows exactly what will change when you commit.
</Step>

<Step title="Commit or rollback">
**To keep the changes:**
```bash
curd workspace commit
```

**To discard the changes:**
```bash
curd workspace rollback
```

After commit, changes are written back to your actual workspace files. After rollback, the shadow workspace is cleaned without touching your files.
</Step>

</Steps>

<Warning>
Always use workspace sessions for mutations. Running `curd edit` without an active session will fail with a barrier error.
</Warning>

## Your First Workflow Script

CURD's `.curd` scripting language lets you author repeatable workflows.

<Steps>

<Step title="Create a script file">
Create a file named `fix_alpha.curd`:

```curd fix_alpha.curd
# explain: Update the alpha function safely
# why: Modernize implementation while preserving API
# risk: alpha is called in multiple modules

use session required

let patch = """
pub fn alpha() {
    println!("Modernized implementation");
}
"""

atomic {
  edit uri="src/lib.rs::alpha" action="upsert" code=$patch
  verify_impact strict=true
}
```
</Step>

<Step title="Check the script">
```bash
curd run check fix_alpha.curd
```

This analyzes the script without executing it and reports:
- Resolved targets
- Graph-adjacent impact
- Conflict risk
- Required safeguards
</Step>

<Step title="Compile to a plan artifact">
```bash
curd run compile fix_alpha.curd
```

This creates a compiled plan artifact under `.curd/plans/` with:
- Bound arguments
- Explainability metadata
- Execution safeguards
</Step>

<Step title="Execute the workflow">
```bash
curd workspace begin
curd run fix_alpha.curd
curd workspace commit
```

The script runs within a workspace session and applies all changes atomically.
</Step>

</Steps>

## Common Workflows

### Read-Only Exploration

Use this pattern when you just want to understand code:

```bash
curd search function_name
curd read file.rs::function_name
curd graph file.rs::function_name --direction up --depth 1
```

### Safe Refactoring

Use this pattern for any code mutation:

```bash
curd workspace begin
curd edit file.rs::old_function --code "new implementation"
curd workspace diff
curd workspace commit  # or rollback
```

### Team/Agent Workflow

Use this pattern for reviewable, governed workflows:

```bash
curd run check script.curd
curd run compile script.curd
curd plan edit <plan-id>  # Review and adjust
curd workspace begin
curd run script.curd
curd workspace commit
```

## Interactive REPL

For exploratory workflows, use the interactive REPL:

```bash
curd repl
```

This starts a persistent session where you can run commands interactively without typing `curd` each time.

## Runtime Modes

CURD has two runtime ceilings:

<CardGroup cols={2}>
  <Card title="Full Mode" icon="infinity">
    All capabilities enabled: search, graph, read, edit, workspace, plan execution, build, MCP server
  </Card>
  <Card title="Lite Mode" icon="lock">
    Restricted to core operations: search, read, edit, graph, and basic workspace commands (status, list, dependencies)
  </Card>
</CardGroup>

Set the mode with `CURD_MODE=lite` or `CURD_MODE=full` (default is full).

<Note>
Runtime ceilings are hard limits. Even if a profile requests more capabilities, lite mode will restrict them.
</Note>
