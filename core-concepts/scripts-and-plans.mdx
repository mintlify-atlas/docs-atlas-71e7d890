CURD separates **authoring** (`.curd` scripts) from **governed execution** (compiled plan artifacts). This split enables readable workflows for developers while providing inspectable, safeguarded execution for teams.

## Why Both Exist

From SCRIPTS_AND_PLANS.md:10:

> ".curd is better for humans and agents to author intent. Compiled plan artifacts are better for bound arguments, explainability metadata, safeguards, profile/runtime context, and repeatable execution."

## .curd Source Language

The `.curd` language is a DSL for authoring code mutation workflows.

### Supported Constructs

From SCRIPTS_AND_PLANS.md:27:

- `use profile ...` — Declare execution profile
- `use session ...` — Declare session requirement
- `arg` — Define script arguments
- `let` — Variable assignment
- Multiline strings (`"""..."""`) — Patch content
- Tool-call statements — Invoke CURD tools
- `sequence` — Sequential execution block
- `atomic` — Transactional block (rollback on failure)
- `abort` — Explicit rollback trigger

<Warning>
`parallel` is parsed but **rejected** at compile/execute time until the runtime has honest parallel lowering.
</Warning>

### Example Script

```curd
# explain: tighten auth validation without changing the public entrypoint
# why: downstream callers depend on the current function name
# risk: auth and session modules are tightly connected

use session required

let patch = """
pub fn validate(token: &str) -> bool {
    !token.is_empty()
}
"""

atomic {
  edit uri="src/auth.rs::validate" action="upsert" code=$patch
  verify_impact strict=true
}
```

### Explainability Comments

From SCRIPTS_AND_PLANS.md:45:

Structured comments are preserved into compiled metadata:

- `# explain: ...` — High-level intent
- `# why: ...` — Rationale
- `# risk: ...` — Known hazards
- `# review: ...` — Review notes
- `# tag: ...` — Categorization tags

These become part of the compiled plan artifact for audit and review.

### Script Arguments

Arguments enable one source script to generate multiple concrete plan artifacts:

```curd
arg target_uri: string
arg strict: bool = true

let patch = """
pub fn alpha() {}
"""

edit uri=$target_uri action="upsert" code=$patch
verify_impact strict=$strict
```

**Compilation with arguments**:

```bash
curd run compile fix.curd --target-uri src/lib.rs::alpha
curd run compile fix.curd --target-uri src/lib.rs::beta --strict false
```

Each invocation produces a different plan artifact with bound argument values.

## Script Lifecycle

<Steps>

### 1. Check

```bash
curd run check fix_auth.curd
```

Validates the script without mutation:
- Resolves target URIs
- Computes graph-adjacent impact
- Detects conflict risk
- Reports session requirement
- Suggests safeguards

From SCRIPTS_AND_PLANS.md:92:

> "This is the step that turns a script from 'idea' into 'something you can reason about'."

### 2. Compile

```bash
curd run compile fix_auth.curd
```

Emits a compiled plan artifact under `.curd/plans/`.

Artifact contains:
- Compiled DSL/plan payload
- Source hash
- Source path
- Bound argument values
- Explainability metadata
- Safeguards
- Runtime ceiling snapshot

### 3. Edit the Plan

```bash
curd plan edit <plan-id>
```

Guided edits for:
- Profile selection
- Default `output_limit`
- Per-node `output_limit`
- Default `retry_limit`
- Per-node `retry_limit`

From SCRIPTS_AND_PLANS.md:128:

> "This is where a team can add stronger execution posture after the authoring step: choose a stricter profile, lower output budgets, reduce retries."

### 4. Execute

For mutating scripts:

```bash
curd workspace begin
curd run fix_auth.curd
curd workspace commit
```

If execution should be discarded:

```bash
curd workspace rollback
```

</Steps>

## Plan Artifacts

Compiled plan artifacts are the execution contract.

### Plan Structure

From curd-core/src/plan/engine.rs:186:

```rust

    pub id: Uuid,
    pub nodes: Vec<PlanNode>,
}

pub struct PlanNode {
    pub id: NodeId,
    pub op: ToolOperation,              // Tool call or internal command
    pub dependencies: Vec<IdOrTag>,     // Dependency DAG
    pub output_limit: usize,            // Artifact compaction threshold
    pub retry_limit: u8,                // Retry attempts
}
```

### Tool Operation Types

```rust

    /// Standard MCP Tool Execution
    McpCall {
        tool: String,
        args: serde_json::Value,
    },
    /// Internal Engine Operations

        command: String,

```

**Allowed internal commands** (curd-core/src/policy.rs:162):
- `recompute_graph`
- `invalidate_index`
- `emit_marker`
- `clear_shadow`

<Note>
`clear_shadow` is **human-only** and cannot be used in agent-executed plans.
</Note>

### Dependency Resolution

Nodes can depend on other nodes via ID or tag:

```rust
// curd-core/src/plan/engine.rs:205
pub enum IdOrTag {
    Id(NodeId),   // Direct UUID reference
    Tag(String),  // Named tag for semantic grouping
}
```

Execution respects the dependency DAG:

```
┌─────────┐
│ Node A  │
│ (edit)  │
└────┬────┘
     │
     ▼
┌─────────┐
│ Node B  │ (depends on A)
│ (verify)│
└────┬────┘
     │
     ▼
┌─────────┐
│ Node C  │ (depends on B)
│ (commit)│
└─────────┘
```

## Compilation Flow

The `.curd` source is parsed into an AST and lowered to a plan:

<Steps>

### Parse Source

Lex and parse `.curd` file into `DslNode` AST

### Validate Syntax

Check for unsupported constructs (e.g., `parallel`)

### Resolve Arguments

Bind provided argument values (e.g., `--target-uri`)

### Lower to Plan

Convert `DslNode` tree to `PlanNode` DAG with dependencies

### Attach Metadata

Include explainability comments, source hash, safeguards

### Emit Artifact

Serialize plan to `.curd/plans/<uuid>.json`

</Steps>

### DSL Node Types

From curd-core/src/plan/engine.rs:61:

```rust
pub enum DslNode {
    /// Tool call: tool_name(args)
    Call {
        tool: String,
        args: serde_json::Value,
    },
    /// Transactional block: atomic { ... }
    Atomic { nodes: Vec<DslNode> },
    /// Deterministic termination: abort("reason")
    Abort { reason: String },
    /// Variable assignment: let x = ...
    Assign {
        var: String,
        value: serde_json::Value,
    },
}
```

## Safeguards and Constraints

### Plan Validation

From curd-core/src/policy.rs:406:

```rust
pub fn validate_plan(
    &self,
    plan: &crate::plan::Plan,
    is_human: bool,
    workspace_root: &Path,
) -> PolicyDecision {
    // 1. Node count limit
    if plan.nodes.len() > MAX_PLAN_NODES {  // 128
        return PolicyDecision::Deny("PLAN_REJECTED: exceeds max node count");
    }

    // 2. Unique node IDs
    if duplicate_ids_exist(plan) {
        return PolicyDecision::Deny("PLAN_REJECTED: duplicate node id");
    }

    // 3. Dependency fan-in limit
    if node.dependencies.len() > MAX_PLAN_DEPENDENCIES {  // 32
        return PolicyDecision::Deny("PLAN_REJECTED: exceeds max dependency fan-in");
    }

    // 4. Retry limit
    if node.retry_limit > MAX_PLAN_RETRY_LIMIT {  // 3
        return PolicyDecision::Deny("PLAN_REJECTED: exceeds max retry_limit");
    }

    // 5. Output limit bounds
    if node.output_limit == 0 || node.output_limit > MAX_PLAN_OUTPUT_LIMIT_BYTES {  // 1 MB
        return PolicyDecision::Deny("PLAN_REJECTED: invalid output_limit");
    }

    // 6. Per-node policy checks
    for node in plan.nodes {
        match self.evaluate(node.tool, node.args, is_human, workspace_root) {
            PolicyDecision::Deny(reason) => {
                return PolicyDecision::Deny(format!("PLAN_REJECTED: {}", reason));
            }
            _ => {}
        }
    }

    PolicyDecision::Allow
}
```

<Warning>
Plan validation is **all-or-nothing**. If any node violates policy, the entire plan is rejected at registration time.
</Warning>

### Retry Policy

## Execution Model

Plans execute with focus tracking and checkpoint persistence:

### Focus State

From curd-core/src/plan/engine.rs:244:

```rust
struct FocusState {
    updated_at_secs: u64,
    active_tools: Vec<String>,      // Recently invoked tools
    active_uris: Vec<String>,       // Recently accessed URIs
    active_symbols: Vec<String>,    // Recently touched symbols
}
```

**Focus pruning** limits context size by removing distant nodes from the graph:

```rust
// curd-core/src/plan/engine.rs:263
struct FocusPrunePolicy {
    enabled: bool,
    max_distance: u8,       // BFS depth from active symbols
    max_uris: usize,        // Hard cap on URI count
    override_keep_uris: Vec<String>,  // Always keep these
}
```

<Info>
This prevents plans from pulling in the entire codebase context, improving performance and reducing token usage for LLM-driven agents.
</Info>

### Checkpoints

- Debug plan execution history
- Audit what was executed and when

## Trust Model

From SCRIPTS_AND_PLANS.md:178:

> "`.curd` files are normal source files. Compiled plan artifacts are the stronger execution objects."

**Implications**:
- Humans and agents author `.curd` scripts (version-controlled)
- CURD compiles and enriches the result (adds safeguards)
- Teams inspect and approve plan artifacts before execution
- Plan artifacts are the governance boundary

<Steps>

### Author Script

Write `.curd` with readable intent

### Compile to Plan

CURD validates and adds safeguards

### Review Plan Artifact

Team inspects compiled JSON for safety

### Execute with Governance

Plan runs with profile/policy enforcement

</Steps>

## When to Use Each

### Stay in .curd Scripts When...

From SCRIPTS_AND_PLANS.md:194:

- You want readable, reusable workflows
- Easier authoring than raw JSON
- Iterative graph-aware preflight
- Parameterized mutation patterns

### Care About Plan Artifacts When...

From SCRIPTS_AND_PLANS.md:202:

- You need stable execution history
- Explicit safeguards
- Inspectable defaults
- Governance-ready execution records

<Note>
For internal agent infrastructure, the plan artifact is the execution object. The script is the authoring source. The gap between them is where safeguards, explainability, and policy become concrete.
</Note>

## Example: Full Workflow

<Steps>

### Write Script

```curd
# file: tighten_auth.curd
# explain: add null check to validate function
# risk: auth module is mission-critical

use profile supervised
use session required

arg target: string = "src/auth.rs::validate"

let patch = """
pub fn validate(token: &str) -> Result<bool> {
    if token.is_empty() {
        return Err(anyhow!("empty token"));
    }
    Ok(true)
}
"""

atomic {
  edit uri=$target action="upsert" code=$patch
  verify_impact strict=true
}
```
